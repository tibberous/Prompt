Overview

Prompt is a desktop workbench for building strong LLM instructions from reusable pieces instead of typing one giant prompt from scratch every time. It combines prompt files, doctypes, workflows, a live prompt output pane, a font tool, settings, help, and an About box into one lightweight app.

Quick Start

Start in the Prompts tab. Choose a starter prompt from the prompt selector, choose a workflow, choose a doctype, fill in the task/context fields, and copy the generated prompt from the output pane. The app is designed for real work: coding, debugging, refactoring, research, documentation, and project handoff prompts.

Main Window Layout

The main window opens with the Prompts workspace. The left side is the prompt runtime/output pane. The right side is the workflow editor/preview pane when enabled. Other tabs provide Doctypes, Fonts, Help, and Settings. The File menu handles open/save/reload actions, and the Help menu opens the About Prompt box.

Prompts Tab

Prompts are markdown files stored in the prompts folder. They are grouped into ordered categories such as Session Bootstrap, Code Analysis and Refactor, Runtime and Debugging, Quality Localization and Data, and Research Docs and Prompt Design. Prompt filenames can use numeric prefixes to control order, but the selector displays clean Title Case names.

Starter Prompt Library

The built-in starter library is meant to make Prompt useful immediately. The strongest repeated practices are included where they belong: think longer for a better answer, preserve important constraints, state what is unfinished, run AST before coding, grep for whole classes of errors, unify to a current working version, and deliver a full zip when appropriate.

Prompt Chaining

When a prompt is generated, the app stores the previous generated prompt so the next prompt can build on it. This allows a large project prompt to be assembled in stages: first bootstrap the session, then analyze code, then refactor, then test, then produce a handoff checklist.

Workflows

Workflows define the interactive controls that appear in the prompt runtime pane. A workflow can include checkboxes, radio groups, text fields, multiline text areas, headings, comments, and explanatory text. The workflow source is markdown-like, then compiled into HTML for the WebEngine preview.

Workflow Markdown Syntax

Use [ ] or [x] for checkboxes, [o], [O], or [0] for radio options, [____] for single-line text inputs, and [__________] for larger note fields. Markdown headings and simple formatting are supported. Descriptive comments can be shown as styled blockquote boxes so workflow authors can explain why a control exists.

Comments and Notes

Prompt supports visible descriptive comments in markdown. Supported styles include HTML comments, hash comments, double-slash comments, block comments, and [comment]...[/comment] blocks. These render as drop-shadowed note boxes in preview instead of disappearing, which makes workflows and doctypes easier to document.

Doctypes

Doctypes are response-preference files. They define how the LLM should behave after the prompt is copied into a model session. A doctype can describe RP / RESPONSE PREFERENCE, WARN, STOP, ASK, INFER, ASSUME, and SKILL blocks. The current starter doctypes are Normal, Cautious, and CodeBot.

Normal Doctype

Normal is the balanced everyday doctype. It is meant for general useful work: answer directly, preserve constraints, ask only when needed, infer obvious details, warn when something matters, and avoid overcomplicating simple requests.

Cautious Doctype

Cautious is for higher-risk work where assumptions can cause expensive mistakes. It raises the threshold for warnings, asks more follow-up questions when ambiguity matters, and treats safety, correctness, legal/medical/financial uncertainty, and irreversible project changes with more care.

CodeBot Doctype

CodeBot is the heavy coding/project doctype. It tells the model to start with AST, keep a high-level architecture view, grep for whole classes of errors, unify the repo into one CWV, delete branch clutter, compile/run/test when possible, and clearly state what was not finished.

Doctype Preview

The Doctypes tab includes an editor and preview. The preview is monospace-friendly and uses Prism.js for fenced code blocks. Code blocks include non-copying line-number gutters and copy controls so handbook/code sections are readable without polluting copied text.

Fonts Pane

The Fonts pane opens bundled fonts from the fonts folder and displays glyphs with their index, glyph name, integer value, and U+ codepoint. Users can add glyphs to a builder list and export subsets in supported font formats. It is useful for UI asset work and icon/font experiments.

Settings

The Settings tab stores database connection fields such as host, port, database, user, and password. Settings are stored through the app's SettingsStore using SQLite by default. Window geometry, splitter positions, panel visibility, and prompt runtime session state are also persisted.

Menus and Commands

The File menu provides Open Prompt, Save Prompt, Save Workflow, Save Doctype, Open Font, and Reload Everything. The Help menu opens About Prompt. The tab strip itself is the primary navigation between Prompts, Doctypes, Fonts, Help, and Settings.

About Box

The About box uses the same WebEngine-style HTML presentation as the rest of the app. It includes Prompt 1.0 version information, Trenton Tompkins contact details, portfolio link, MIT License link, Chat GTP 5.4 Thinking credit, the cloud artwork, and the spry Tamagotchi.

Launcher and Debugger

start.py is the launcher/debugger layer. It handles dependency checks, offscreen/capture flags, trace flags, debugger surfaces, heartbeat/fault tables, and child process launch. The app itself lives in prompt_app.py, while start.py owns bootstrap and runtime supervision.

Offscreen and Screenshot Testing

The launcher supports --offscreen and verbose trace modes intended for headless testing and screenshot capture. In a correctly provisioned environment, this can start the app in a managed display, write trace logs, and save screenshots for documentation or debugging.

WebEngine Diagnostics

Preview panes use a logging WebEngine wrapper when Qt WebEngine is available. Console messages, load events, URL changes, render-process termination, source viewing, and context menu behavior are routed through shared code so HTML/CSS/JS issues are easier to diagnose.

Project Files and Folders

Important folders include prompts for starter prompt markdown files, doctypes for response-preference/handbook files, workflows for interactive prompt builders, vendor for bundled JavaScript/CSS like jQuery and Prism, fonts for font assets, assets for artwork, generated for runtime HTML output, and logs for runtime diagnostics.

Saving and Reloading

Saving a prompt writes the current prompt document to the prompt library. Saving a workflow writes the workflow markdown. Saving a doctype writes the selected doctype. Reload Everything reloads workflows, doctypes, prompts, fonts, settings help, previews, and runtime output from disk.

Recommended Workflow

For coding sessions, start with Initial Response Preferences or Stage New Zip and Start New Project. For bug fixing, use a runtime/debugging prompt. For documentation, use a docs/review prompt. For high-risk changes, use the Cautious doctype. For code handoffs, use CodeBot.

Troubleshooting

If a preview does not update, use Reload Everything. If the WebEngine output looks stale, check generated HTML files. If offscreen capture fails, confirm PySide6 and Linux display tools are installed or bundled. If a prompt is not visible, check the prompts folder, filename extension, and numeric sorting prefix.

Release Readiness

Before packaging, run a compile check, launch through start.py, test the main tabs, verify the help page, open About Prompt, confirm starter prompts/doctypes are present, clear stale generated files if needed, and package the unified CWV. The next planned release step is creating a Windows executable and installer.

CodeBot Handoff Pattern

CodeBot prompts are designed for serious project sessions where the model must inspect first, change carefully, test, and tell the truth about unfinished work.

Hit it with AST first.
Keep the high-level architecture in memory.
Grep for the whole class of error before fixing one instance.
Unify into one CWV.
Delete duplicate working files.
Run compile/runtime checks when possible.
Deliver a complete zip.
State exactly what is not finished.
